    1.1. Fundamentos de fetch, promesas y async/await

    Antes de poder trabajar con datos abiertos (CSV, JSON, XML o APIs), necesitamos saber cómo traer esos datos a la web usando JavaScript. El navegador nos proporciona la función fetch, que permite realizar solicitudes HTTP de forma sencilla y moderna.

    En esta lección vamos a detenernos en qué es fetch, qué papel juegan las promesas, qué ventajas ofrece async/await y qué limitaciones debemos tener en cuenta al trabajar con este enfoque.

    📌 ¿Qué es fetch?

    fetch(url) es una función nativa de JavaScript que nos permite pedir recursos a un servidor. Devuelve una promesa que se resuelve en un objeto Response, con métodos para acceder al contenido.

    Ejemplo básico:

    fetch("datasets/csv/hosteleria-a-domicilio.csv")
  .then(res => res.text())
  .then(texto => console.log(texto))
  .catch(err => console.error("Error:", err));

    En este caso:

    • res.text() obtiene el archivo como texto.
    • Usamos .then() para manejar el resultado cuando llega.
    • Si algo falla (por ejemplo, la ruta es incorrecta), el .catch() captura el error.

    📌 ¿Qué es una promesa?

    Una promesa es un objeto que representa un valor que aún no conocemos pero que estará disponible en el futuro.

    Estados de una promesa:

    1. pending → pendiente.
    2. fulfilled → resuelta con éxito.
    3. rejected → rechazada por error.

    Las promesas nos permiten no bloquear la página mientras esperamos datos del servidor.

    📌 async y await

    En lugar de encadenar muchos .then(), podemos escribir código más claro con async/await.

    Ejemplo equivalente al anterior:

    async function cargarCSV() {
  try {
    let res = await fetch("datasets/csv/hosteleria-a-domicilio.csv");
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
    let texto = await res.text();
    console.log(texto);
  } catch (err) {
    console.error("Error:", err);
  }
}

    Ventajas de este enfoque:

    • El código se lee “de arriba a abajo”, como si fuera síncrono.
    • Más fácil de mantener en programas largos.
    • El try/catch captura los errores de forma ordenada.

    📌 Particularidades al trabajar con fetch

    • Protocolos seguros: algunos navegadores bloquean peticiones desde http:// si la página está en https://.
    • CORS: el servidor debe permitir que otros sitios consuman sus datos (algo que el portal de Datos Abiertos de JCyL ya hace).
    • Rutas relativas: asegúrate de que apuntas a la carpeta correcta dentro del proyecto.
    • Servidor local: abrir archivos con file:// no funciona; necesitas un servidor local (mkdocs serve, Live Server, etc.).

    📌 Ventajas de usar fetch

    • API nativa: no requiere instalar librerías.
    • Sencillez: se adapta a la mayoría de casos de uso.
    • Flexible: permite trabajar con texto, JSON, blobs, etc.
    • Compatible: funciona en todos los navegadores modernos.

    📌 Limitaciones y retos

    • No aborta por defecto: necesitamos AbortController si queremos cancelar una petición.
    • Sin soporte directo para timeout: hay que implementarlo manualmente.
    • Gestión de errores limitada: solo da error de red; para códigos como 404 o 500 debemos comprobar res.ok.
    • Complejidad con peticiones encadenadas: sin async/await, los .then() pueden volverse difíciles de leer.

    Consejo

    Cuando trabajes con datos abiertos:

    • Usa siempre try/catch o .catch() para manejar errores.
    • Comprueba res.ok para detectar respuestas HTTP incorrectas.
    • Si los datos son muy pesados, considera paginarlos o mostrarlos poco a poco.
